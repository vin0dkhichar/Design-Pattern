🔹 Helm + Kubernetes

🔹 Next.js + FastAPI
Q: What is Next.js?
→ A React framework for building frontend apps with SSR/CSR/ISR support.
Q: What is FastAPI?
→ A Python backend framework for building high-performance APIs.
Q: What is ASGI?
→ Async Server Gateway Interface that allows handling multiple requests concurrently.
Q: Why async/await in FastAPI?
→ Enables non-blocking I/O, improving performance for concurrent requests.
Q: What is automatic OpenAPI docs?
→ FastAPI auto-generates API docs (Swagger) from code and type hints.
Q: What is SSR?
→ Server renders HTML on each request.
Q: What is CSR?
→ Browser renders UI using JavaScript.
Q: What is ISR?
→ Static pages that update periodically without full rebuild.
“SSR renders pages on the server and sends fully prepared HTML to the client, which improves SEO and initial load performance. CSR, on the other hand, renders content in the browser using JavaScript, which gives better interactivity but slower first load. The main trade-off is between SEO and initial performance vs client-side flexibility and reduced server load.”



🔹 OIDC (OpenID Connect)
Q: What is OIDC?
→ An authentication protocol built on OAuth2 that provides user identity.
Q: What is the OIDC flow?
→ Redirect → login → authorization code → exchange for tokens.
Q: What is an ID token?
→ A JWT that contains user identity information.
Q: What is an access token?
→ A token used to access protected APIs.
Q: What is a refresh token?
→ Used to get new tokens without logging in again.
Q: What is JWT validation?
→ Verifying signature, expiry, issuer, and audience.

🔹 OAuth 2.0
Q: What is OAuth 2.0?
→ An authorization framework that allows apps to access user resources without sharing passwords.
Q: Simple example?
→ “Login with Google” → app gets access to your profile without knowing your password.
🔹 Key Idea
Q: What problem does OAuth solve?
→ Securely delegate access to third-party apps.
🔹 Roles in OAuth
Q: Who are the main roles?
→ Resource Owner → User
→ Client → Your app
→ Authorization Server → Auth provider (Google, eSignet)
→ Resource Server → API server
🔹 Tokens
Q: What is an access token?
→ A token used to access protected APIs.
Q: What is a refresh token?
→ Used to get a new access token when it expires.
🔹 OAuth Flow (Authorization Code Flow – MOST IMPORTANT)
Q: What is the flow?
Client redirects user → Authorization Server
User logs in and gives permission
Server returns authorization code
Client exchanges code → access token
Client calls API using access token
🔹 Why Authorization Code Flow?
Q: Why is it secure?
→ Token is exchanged on backend, not exposed in browser.
🔹 Other Flows (basic idea)
Q: Implicit flow?
→ Token returned directly (less secure, mostly deprecated)
Q: Client Credentials flow?
→ Machine-to-machine communication (no user)
Q: Password flow?
→ User shares password directly (not recommended)
🔹 OAuth vs Authentication
Q: Does OAuth authenticate users?
→ No, it only authorizes access
Q: Then how do we get user identity?
→ Use OIDC (OpenID Connect) on top of OAuth
🔹 Where JWT comes in
Q: Is access token always JWT?
→ Often yes, but can also be opaque tokens.
Q: What does JWT contain?
→ Claims like user ID, permissions, expiry.
🔹 Security Concepts
Q: What is scope?
→ Defines what access is allowed (e.g., read profile).
Q: What is redirect URI?
→ URL where auth server sends response (must be registered).
Q: What is state parameter?
→ Prevents CSRF attacks.
🔹 Real Flow Example
Q: Example in your project (eSignet)?
→ User → eSignet login → returns code → backend exchanges → gets tokens → API access

Q: Storing access token insecurely?
→ Risk of token theft
Q: Using OAuth instead of OIDC for login?
→ Missing identity layer

“OAuth 2.0 is an authorization framework that allows applications to access user resources without sharing credentials. It works by issuing access tokens after user consent. In the authorization code flow, the user is redirected to an authorization server, logs in, and the client exchanges a code for an access token. OAuth itself handles authorization, and for authentication, it is extended using OpenID Connect.”


🔹 SSO (Single Sign-On)
Q: What is SSO?
→ Login once and access multiple applications without re-authentication.
Q: How does SSO work?
→ User logs in via IdP → gets token/session → reused across apps.
Q: What is an IdP?
→ Identity Provider that authenticates users (e.g., eSignet).
Q: Session-based SSO vs Token-based SSO?
→ Session uses cookies; token-based uses JWT.
Q: SSO vs OIDC?
→ SSO is concept; OIDC is protocol to implement it.

🔹 Authentication
Q: What is authentication?
→ Process of verifying who the user is.
Q: Example?
→ Logging in with username & password.
Q: What does it check?
→ Identity (are you really Vinod?)
Q: Common methods?
→ Passwords, OTP, biometrics, OAuth/OIDC login
Q: Output of authentication?
→ Identity proof (session or ID token)
🔹 Authorization
Q: What is authorization?
→ Process of deciding what the user can access.
Q: Example?
→ Admin can delete users, normal user cannot.
Q: What does it check?
→ Permissions/roles
Q: Common methods?
→ RBAC (Role-Based Access Control), scopes, policies
Q: Output of authorization?
→ Access granted or denied

Q: Authentication vs Authorization?
→ Authentication = Who you are
→ Authorization = What you can do
🔹 Order of Flow
Q: Which comes first?
→ Authentication → then Authorization
Q: Why?
→ System must know who you are before deciding permissions.
🔹 Real Example (Simple)
Q: Example in app?
→ Login → authentication
→ Access dashboard → authorization
🔹 Real Example (OAuth/OIDC)
Q: Where does each fit?
→ OIDC → authentication (identity)
→ OAuth → authorization (API access)
🔹 Token Mapping
Q: Which token is for what?
→ ID Token → authentication
→ Access Token → authorization
🔹 Common Mistakes
Q: Using only authentication?
→ User logged in but no permission control
Q: Mixing both concepts?
→ Leads to security issues

“Authentication is the process of verifying a user’s identity, such as logging in with credentials, while authorization determines what actions that authenticated user is allowed to perform. Authentication comes first, followed by authorization. For example, OpenID Connect handles authentication by providing identity tokens, while OAuth handles authorization using access tokens.”

🔹 React Fiber
🔸 Core Idea
Q: What is React Fiber?
→ A new reconciliation engine in React that makes rendering incremental and interruptible.
Q: Why was Fiber introduced?
→ Old React (stack-based) blocked the UI during rendering → poor performance for large apps.
🔹 Problem Before Fiber
Q: What was the issue with old reconciliation?
→ Rendering was synchronous (all at once) → UI freeze.
🔹 Fiber Solution
Q: What does Fiber change?
→ Breaks rendering into small units of work (fibers).
Q: What is a Fiber node?
→ A JavaScript object representing a component and its work.
🔹 How Fiber Works
🔸 1. Work Splitting
Q: What is work splitting?
→ Large rendering task is divided into smaller chunks.
🔸 2. Interruptible Rendering
Q: What does interruptible mean?
→ React can pause rendering to handle high-priority tasks (like user input).
🔸 3. Scheduling
Q: What is scheduling?
→ React decides which updates to process first based on priority.
🔹 Two Phases of Fiber
🔸 1. Render Phase (Reconciliation)
Q: What happens here?
→ Builds new Fiber tree (Virtual DOM)
→ Can be paused/resumed
🔸 2. Commit Phase
Q: What happens here?
→ Applies changes to real DOM
→ Always synchronous (fast)
🔹 Fiber Tree Structure
Q: How is Fiber stored?
→ As a linked structure:
parent
child
sibling
Q: Why linked structure?
→ Enables traversal and pausing work efficiently.
🔹 Priority Concept (VERY IMPORTANT)
Q: How does React prioritize updates?
→ Assigns priority levels (lanes)
Q: Example?
→ User typing → high priority
→ Data fetching → low priority
🔹 Key Benefits
Q: Why Fiber is powerful?
→ Smooth UI (no blocking)
→ Better responsiveness
→ Enables concurrent features
🔹 Example (Conceptual)
Q: What happens when typing in input?
→ React pauses heavy rendering → updates input instantly → resumes work later
🔹 Fiber vs Old React
Feature	Old React	Fiber
Rendering	Synchronous	Incremental
Interruptible	❌	✅
Scheduling	❌	✅

Q: Why commit phase is not interruptible?
→ To avoid inconsistent UI (DOM must stay stable).

“React Fiber is the new reconciliation engine that enables incremental and interruptible rendering. Unlike the old synchronous approach, Fiber breaks rendering into small units of work and schedules them based on priority. It has two phases: a render phase that builds the fiber tree and can be paused, and a commit phase that applies changes to the DOM synchronously. This improves responsiveness and enables features like concurrent rendering.”



🔹 Nginx
Q: What is Nginx?
→ A web server and reverse proxy that handles incoming requests and forwards them to backend services.
Q: What does Nginx do in backend systems?
→ Load balancing, SSL termination, caching, routing requests to services.
Q: What is a reverse proxy?
→ Nginx sits in front of backend servers and forwards client requests to them.
🔹 Kubernetes Ingress
Q: What is Ingress?
→ A Kubernetes resource that manages external access (HTTP/HTTPS) to services.
Q: What is an Ingress Controller?
→ The actual implementation (often Nginx) that processes Ingress rules.
Q: How do Nginx and Ingress relate?
→ In Kubernetes, Nginx runs as Ingress Controller to route traffic to services.
Q: Example flow?
→ Client → Nginx Ingress → Service → Pod
🔹 Large Headers Issue
Q: What are HTTP headers?
→ Metadata sent with requests (cookies, auth tokens, etc.)
Q: What is “large header” issue?
→ When request headers exceed Nginx buffer limits.
Q: Common error?
→ 400 Bad Request - Request Header Too Large
Q: Why does it happen?
→ Headers (especially cookies or JWTs) exceed configured buffer size.
Q: Default Nginx limits?
→ Typically ~4KB–8KB per header buffer.
🔹 Cookie Size Problem
Q: Why do cookies cause large headers?
→ Cookies are sent in every request header.
Q: Max cookie size?
→ ~4KB per cookie.
Q: Why is this a problem?
→ Multiple cookies → header becomes very large.
Q: Real-world example?
→ Storing large JWT or user data in cookies → exceeds limits.
🔹 Nginx + Ingress Failure Scenario
Q: What happens in production?
→ Client sends request with large cookies
→ Nginx Ingress rejects request
→ User gets 400 error
Q: Why specifically in Ingress?
→ Ingress uses Nginx → same buffer limits apply.
🔹 Fixes (Nginx / Ingress)
Q: How to increase header size in Nginx?
large_client_header_buffers 4 16k;
Q: In Kubernetes Ingress (Nginx)?
nginx.ingress.kubernetes.io/proxy-buffer-size: "16k"
nginx.ingress.kubernetes.io/large-client-header-buffers: "4 16k"
Q: What do these do?
→ Increase allowed header size to avoid rejection.
🔹 Best Practices (VERY IMPORTANT)
Q: Should we store large data in cookies?
→ No
Q: What to store instead?
→ Only small identifiers (session ID, short token)
Q: Better alternative?
→ Store data in backend (DB/Redis) and reference via ID.
Q: JWT best practice?
→ Keep JWT small OR store in Authorization header instead of cookies.
🔹 Interview-Level Insight
Q: Why large headers are dangerous?
→ Increase latency + memory usage + can break proxies like Nginx.

“Nginx acts as a reverse proxy and is commonly used as an Ingress controller in Kubernetes to route external traffic to services. Large header issues occur when request headers, especially cookies or JWT tokens, exceed Nginx’s buffer limits, leading to errors like ‘400 Request Header Too Large’. Since cookies are sent with every request and limited to around 4KB each, storing large data in cookies can quickly exceed limits. This can be mitigated by increasing Nginx buffer sizes, but the best practice is to keep cookies small and store data in backend systems instead.”
